# Positioning

GlassGradients is not a component library and not a visual toy.
It is a surface engine.

## What that means

It focuses on the part most packages leave weak:

- production-ready glass surfaces
- separated `surface`, `gradient`, and `frost` modes
- recipes for real UI roles
- responsive and `prefers-color-scheme` variants in the same config model
- CSS compile and runtime driven by the same config model
- readable defaults for real text, forms, dialogs, sidebars, and product shells
- graceful fallback when `backdrop-filter` is weak or unavailable
- theme tokens and SSR helpers for framework integrations

## What it is not trying to be

- not a React-only design system
- not a Tailwind-only utility pack
- not a one-shot generator that looks good only in demos
- not a WebGL-only experiment with no fallback story

## Practical differentiation

Compared with lighter CSS-only packages, GlassGradients gives you layered surfaces, motion controls, recipes, responsive variants, and runtime parity.

Compared with heavier systems such as full component libraries, it stays framework-agnostic and keeps the API small.

Compared with flashy liquid-glass demos, it prioritizes readable surfaces, performance profiles, browser fallback, and compatibility.
